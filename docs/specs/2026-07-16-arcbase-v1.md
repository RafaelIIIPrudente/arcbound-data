# Spec: ArcBase v1 — LinkedIn post-metrics dashboard on the template

Status: ready-for-agent · Source of truth: this spec +
[SRS `docs/SRS/SPEC.md`](../SRS/SPEC.md) + the design brief
(`docs/arcbase-dashboard-design-brief/`) + [CONTEXT.md](../../CONTEXT.md) +
[ADR 0006](../adr/0006-app-owned-posts-table.md) +
[ADR 0007](../adr/0007-arcbase-single-tenant.md).

## Problem statement

This repository is a general multi-tenant B2B SaaS starter, but the product to
build is **ArcBase** — an internal Arcbound tool to register Clients, ingest
scraped LinkedIn post metrics, and view analytics (SRS §1). None of the ArcBase
domain exists in code yet; only the template's Customers reference feature, a
`team`/`roles` demo, and a multi-tenancy migration. This spec specializes the
template into ArcBase and sequences the build into executer handoffs.

## Solution

An internal, auth-gated dashboard. Authenticated Arcbound staff register
**Clients** (individual LinkedIn profiles), upload a weekly **Scrape** (CSV or
JSON) per Client on `/upload`, and view per-Client and aggregate analytics. Each
upload is a first-class immutable **Upload** record; post metrics are upserted
into an app-owned **Posts** table keyed on `linkedin_post_id`. Everything is
single-tenant; RLS enforces "authenticated may read/insert" and immutability.
Deep time-series analytics stays downstream in Shay's Power BI.

## Graded decisions (from the grill, 2026-07-16)

1. **Single-tenant.** Retire Organizations/Memberships/switcher/invites/Org-Roles
   and the `team`/`role-settings` demo. Auth collapses to authenticated-vs-not.
   ([ADR 0007](../adr/0007-arcbase-single-tenant.md))
2. **Clients = repurposed Customers.** Rename the Customers reference feature to
   Clients (table, service, routes, tests); fields `{name, linkedin_url}`; strip
   edit/delete for immutability.
3. **Client identity = normalized `linkedin_url`.** Validate as a
   `linkedin.com/in/…` profile; normalize (lowercase host, force `https`, drop
   `www`, trailing slash, query/hash) before an exact-match uniqueness check
   (resolves OI-01; the design's name-dedup is wrong). A Client is a person.
4. **App owns the Posts table, name configurable.** `public.posts` via migration;
   `SUPABASE_STAGING_TABLE` names it; prod can repoint at Shay's table
   ([ADR 0006](../adr/0006-app-owned-posts-table.md)). Resolves OI-04 (read this
   table directly).
5. **Derive `posted_at`, set-once.** `post_date` is a relative string ("4d");
   compute `posted_at timestamptz` (~day precision) from `post_date` + `scraped_at`
   at ingest, **set on first insert and preserved on later upserts**; keep raw
   `post_date` for audit.
6. **Metric fidelity.** Store the scraped `engagement_rate` as-is (LinkedIn's own
   figure); store `saves` as `NULL` when blank (analytics `COALESCE` to 0).
7. **Ingestion via a transactional Postgres RPC.** `ingest_metrics(...)` upserts
   posts, computes inserted/updated/unchanged in SQL, inserts the `uploads` row,
   and returns counts — one transaction (invariant #4; SRS §7).
8. **Format-type review is optional & skippable.** Keep the review UI but never
   block; unknown Format Type stored as `NULL`, badge shows "—" (relaxes OI-02,
   which would otherwise fire for every post since the scrape leaves it blank).
9. **Latest-snapshot only.** Pure upsert (no per-scrape metric history). "Impressions
   over time" = current metrics bucketed by `posted_at`; KPI deltas = this posting
   period vs the prior; follower trend from the per-Upload history. Growth curves
   are Power BI's job.
10. **Full SRS reshape of the app surface.** Remove the public marketing landing
    and self-serve signup; `/login` (rename sign-in), keep password reset; flat
    routes `/`, `/clients`, `/clients/[id]`, `/upload`, `/resources` with `/` as
    the auth-gated analytics home; accounts provisioned by an admin in Supabase.
11. **Full SRS v1, dependency-sequenced** across six executer handoffs (below).

### Stated defaults (locked)

- Uploader "By" (OI-06): `uploads.uploaded_by = auth.uid()`; display
  `user_metadata.full_name` else email local-part.
- Resources (OI-05): real table `{id, title, url, created_at}`, global, view+add
  only (immutable).
- Post↔Client link: posts attach to the UI-selected `client_id`; `post_name`
  (author) stored raw, not used for linking or hard-validated.
- Dashboard KPIs follow the design comp: Impressions hero + Likes/Comments/
  Reposts/Saves cards + Engagement-rate as its own chart (all six metrics present).
- Branding: "Web App Template"/"Acme" → **ArcBase / Arcbound**; Inter Tight +
  Geist + Geist Mono; red accent `#F63A3A`; light default + dark toggle; `STAGING`
  env tag.
- Config/security: `SUPABASE_STAGING_TABLE`; service-role key server-only;
  follower count required + numeric; recompute counts server-side.

## Data model (app-owned)

```sql
-- clients (repurposed from customers)
id uuid pk, name text not null, linkedin_url text not null, created_at timestamptz
-- uniqueness on normalized linkedin_url

-- uploads (first-class, immutable)
id uuid pk, client_id uuid fk→clients not null, uploaded_at timestamptz,
source_type text check in ('csv','json'), inserted_count int, updated_count int,
unchanged_count int, follower_count int not null, uploaded_by uuid fk→auth.users
-- index (client_id, uploaded_at desc)

-- posts (app-owned; identifier via SUPABASE_STAGING_TABLE; upsert on linkedin_post_id)
linkedin_post_id text pk, client_id uuid fk→clients, urn text, post_url text,
analytics_url text, post_name text, post_content text,
post_date text /*raw relative*/, posted_at timestamptz /*derived, set-once*/,
impressions int, likes int, comments int, reposts int,
engagement_rate numeric, saves int null,
post_format_type text null check in ('image','carousel','link','text','video'),
scraped_at timestamptz

-- resources
id uuid pk, title text not null, url text, created_at timestamptz
```

Confirmed scrape schema (15 cols, in order):
`linkedin_post_id, urn, post_url, analytics_url, post_name, post_content,
post_date, impressions, likes, comments, reposts, engagement_rate, saves,
post_format_type, scraped_at` (verified against
`docs/arcbase-dashboard-design-brief/project/uploads/linkedin_posts_1784129139895.csv`;
the other sample CSV is empty). CSV parsing must honor RFC-4180 quoting
(`post_content` has commas/quotes/newlines).

## RLS

- `clients`, `uploads`, `resources`: authenticated `select` + `insert`; **no**
  update/delete policies.
- `posts`: authenticated `insert` + `update` (upsert only); no delete. Written
  only via `ingest_metrics`.

## Sequenced delivery (one executer handoff each, re-ground between passes)

- **T1 — Reshape & rebrand.** Strip multi-tenancy (drop `init_multitenancy`
  migration + orgs/memberships/invites/switcher/org-authz), remove `team`,
  `role-settings`, `blank`, marketing landing, and self-serve signup; adopt flat
  routes + `/login`; rebrand shell/nav to ArcBase (Dashboard / Client List / Add
  LI Post Metrics / Resources), fonts, accent, theme. Green build, auth still works.
- **T2 — Data model.** Migrations for `clients`, `uploads`, `posts` (configurable
  name), `resources`; RLS above; the `ingest_metrics` RPC (upsert + counts + upload
  row, one tx); `supabase gen types`; env `SUPABASE_STAGING_TABLE`.
- **T3 — Clients feature.** `/clients` list + Add-Client modal (normalized-URL
  validation + dup block) + `/clients/[id]` detail (Uploads/Posts/Followers KPIs +
  upload-history table). UC-02/03/07. (Detail's history is empty until T4 runs.)
- **T4 — Ingestion CORE.** `/upload` (client select, CSV/JSON, follower count,
  format review skippable) → `ingestMetrics` server action → `ingest_metrics` RPC →
  result summary. `papaparse`; relative-date→`posted_at`; all-or-nothing batch. UC-04.
- **T5 — Analytics dashboard.** `/` KPI cards + impressions-over-time + engagement
  chart + recent-posts table + client/date-range filters; empty state → `/upload`.
  UC-05.
- **T6 — Resources + cross-cutting.** `/resources` list+add (UC-06); loading/empty/
  error states everywhere; a11y (focus/keyboard/contrast/reduced-motion); responsive;
  test pass.

## Testing decisions

Assert external behavior; fewest, highest seams.
1. **Unit (Vitest):** `linkedin_url` normalization + dup detection; relative
   `post_date` → `posted_at` parsing; input validation (E1/E2). Prior art:
   `src/lib/authz.test.ts`, `src/services/customers.test.ts`.
2. **Integration vs local Supabase (`supabase start`):** RLS allows authenticated
   read/insert and **blocks update/delete** on `clients`/`uploads` (immutability);
   `ingest_metrics` is atomic (no upload row without posts and vice-versa) and its
   inserted/updated/unchanged counts are correct, including re-upload → 0 inserted.
   Prior art: `src/services/customers.rls.integration.test.ts`.
3. **E2E (Playwright):** login → add client → upload the sample CSV → see the
   result summary → client detail shows the Upload and KPIs. Prior art:
   `e2e/auth-smoke.spec.ts`.

## Out of scope

- The LinkedIn scraper/bookmarklet, Supabase views, Power BI, credential/VPN
  provisioning (external — SRS §12).
- Per-scrape metric history / in-app growth curves (decision 9 — Power BI's job).
- Multi-tenancy, invitations, role tiers (ADR 0007).
- Reconciling our `posts` schema with Shay's real table (a production-cutover
  runbook item, not a build item — ADR 0006).

## Notes

- Delivery is planner → executer: each of T1–T6 is one self-contained executer

- Invariants to preserve every pass (SRS §2): secrets server-side only; Clients &
